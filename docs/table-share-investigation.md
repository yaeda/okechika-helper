# Table Share Investigation

## 目的

変換テーブルを、CSV ファイルを介さずに共有できるかを検討した。

検討対象は次の 3 案。

- URL に変換テーブル本体を埋め込む
- 桶地下サイトなどの HTTPS URL に変換テーブル本体を埋め込む
- 共有用 URL を QR コード化する

結論として、現時点ではどの案も実運用には向かないため、不採用とする。

## 案1: URL に変換テーブル本体を埋め込む

### 想定方式

変換テーブルを次の手順で文字列化して URL に載せる。

1. `target` が `?` または空文字のエントリを除外する
2. `{ version: 1, mappings }` を JSON 化する
3. UTF-8 バイト列に変換する
4. gzip で圧縮する
5. base64url 化する
6. `options.html?...` のクエリまたはハッシュに載せる

### 長さの見積もり

定義済み文字は `1400` 文字、数字系を含めると合計 `1470` 件ある。

`1400` 件を共有する場合の `table` 文字列長の目安は次の通り。

- 変換後が全て 1 文字で同じ値: 約 `4002` 文字
- 変換後が 1 文字である程度ばらつく: 約 `4826` 文字
- 変換後が 2 文字でばらつく: 約 `5015` 文字
- 変換後が 4 文字でばらつく: 約 `5519` 文字

`1470` 件を共有する場合の目安は次の通り。

- 変換後が全て 1 文字で同じ値: 約 `4267` 文字
- 変換後が 1 文字である程度ばらつく: 約 `5191` 文字
- 変換後が 2 文字でばらつく: 約 `5416` 文字
- 変換後が 4 文字でばらつく: 約 `5935` 文字

URL 全体では、これにスキームやパスなどの前置きがさらに加わる。

### 問題点

- `chrome-extension://...` は一般的なチャットや SNS でリンクとして認識されにくい
- 拡張機能 ID が環境依存のため、送信者と受信者でそのまま使えない可能性がある
- ブラウザ単体では長い URL を扱えても、共有経路では 2,000 文字前後を超えると不安定になりやすい

そのため、「送って踏むだけ」の共有リンクとしては適さない。

## 案2: HTTPS の URL に変換テーブル本体を埋め込む

桶地下サイトや、この拡張機能の GitHub リポジトリ / GitHub Pages など、既存の HTTPS サイトを共有先 URL の受け口として使う案を想定する。

### 利点

- 通常の HTTPS リンクとして共有しやすい
- `#...` のハッシュに載せればサーバーへ送られない
- 新しい専用ドメインや専用サービスを用意せずに試せる

### 問題点

- 長さの問題は残る
- サイト側がハッシュを別用途で使っていると競合する
- 拡張未導入環境では受け取っても活用できない
- 桶地下サイトや GitHub 側に本来存在しない共有用途の URL を前提にするため、導線や挙動が不自然になりやすい

`chrome-extension://...` よりは実用的だが、長いデータをそのまま埋め込む方式としてはまだ厳しい。

## 案3: QR コード化

QR コードは、短い共有 URL を渡す用途には向いているが、長いデータ本体をそのまま入れる用途には向かない。

標準 QR コードの最大容量は、Version 40 / 誤り訂正 L でもおおむね次の範囲に収まる。

- Numeric: `7089` 文字
- Alphanumeric: `4296` 文字
- Byte: `2953` bytes

今回の共有文字列は、英数字以外の記号を含む長い URL またはバイト列相当として扱う必要があり、実質的には `Byte` モード寄りになる。

そのため、`4KB` から `6KB` 級の共有文字列は 1 枚の通常 QR コードには収まらない。

加えて、QR コードは主にスマートフォンで読み取る共有導線だが、この拡張機能はモバイル対応していないため、読み取った先でそのまま利用しづらい。

## 代替案

現実的な代替案は次の通り。

- CSV ファイルのエクスポート / インポートを使う
- CSV テキストをクリップボード経由でコピー / ペーストする
- 共有 ID や短い URL を返すための外部 Web サービスを別途用意する

## 結論

この拡張単体で完結させる前提では、URL や QR コードへ変換テーブル本体を直接埋め込む方式は採用しない。

共有導線を強化する場合は、まずは CSV またはクリップボード共有を優先候補とし、真に「リンク共有」を実現したい場合のみ外部の HTTPS サービス導入を再検討する。
